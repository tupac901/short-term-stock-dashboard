# 股票综合分析仪表盘设计

日期：2026-06-28

## 目标

构建一个面向短线和波段交易研究的可部署全栈 Web 投研仪表盘。用户登录后，可以创建个人或团队股票池，配置短线选股策略，系统从公开数据源获取行情和基础数据，计算可解释综合评分，展示排行榜、个股详情、历史评分，并支持准实时盯盘提醒和回测。

第一版强调完整投研闭环，而不是交易执行。系统预留实盘账户连接入口和数据模型，但不接入真实券商账户、不自动交易、不下单。

## 第一版范围

包含：

- 多用户登录和注册
- 团队空间：创建团队、邀请成员、共享股票池和策略
- 股票池管理：手工输入代码，按指数、行业、市场条件生成
- 策略配置：技术面、基本面、资金面、风险项权重
- 综合评分：总分、分项分、入选原因、风险提示
- 选股结果：排行榜、筛选、排序、导出
- 个股详情：价格、指标、评分拆解、历史评分走势
- 准实时盯盘：定时刷新股票池行情，触发提醒
- 提醒中心：价格、涨跌幅、评分变化、指标突破
- 回测系统：策略版本、历史区间、收益、回撤、胜率、交易明细
- 实盘账户连接占位：模块入口、状态字段、未来扩展数据结构

不包含：

- 自动交易
- 券商下单
- 真实实盘账户连接
- 收费订阅和支付
- 复杂团队权限体系

## 实施阶段

第一版按三个里程碑交付，避免一次性实现全部模块导致风险过高。

### 里程碑 1：投研基础闭环

- 登录、注册、用户会话
- 个人股票池
- 手工股票代码输入
- 基础策略配置
- 数据源适配层
- 综合评分排行榜
- 个股评分拆解

### 里程碑 2：团队和盯盘

- 团队创建和成员邀请
- 团队共享股票池和策略
- 准实时行情刷新
- 提醒配置和提醒事件
- 选股结果导出

### 里程碑 3：回测和历史分析

- 策略版本管理
- 评分快照历史
- 回测任务
- 回测报告
- 回测结果对比入口

## 用户与协作

系统支持个人用户和基础团队空间。

用户可以：

- 注册、登录、退出
- 创建个人股票池和策略
- 创建团队
- 邀请成员加入团队
- 在团队内共享股票池和策略

第一版团队权限保持简单：团队创建者管理团队和邀请，成员可以查看和使用团队共享资产。细粒度角色、只读权限、审计权限放到后续版本。

团队邀请第一版采用邀请链接或邀请码，不要求邮件发送能力。邀请加入后，成员可以访问团队共享股票池和策略，但不能删除团队空间。团队创建者可以移除成员。

## 页面结构

### 登录和注册

提供邮箱、密码登录和注册。第一版使用标准密码登录，保留以后接入第三方登录的接口边界。

### 首页仪表盘

展示今日评分概览、Top 股票、风险提示、最近运行记录、活跃提醒和最近回测结果。

### 股票池管理

支持两种创建方式：

- 手工输入股票代码列表
- 按指数、行业、市场、成交额、涨跌幅等条件生成初始股票池

股票池可属于个人或团队。

### 策略配置

用户可以配置评分权重：

- 技术面权重
- 基本面权重
- 资金面权重
- 风险扣分权重

第一版默认策略以短线为主，重点关注趋势、量能、突破、换手、成交额、波动和风险控制。基本面主要作为排雷和过滤项，不作为短线入选的核心依据。

策略保存为版本化配置。每次评分或回测绑定具体策略版本，保证历史结果可追溯。

### 选股结果

展示综合排行榜。每行显示股票代码、名称、综合分、分项分、关键触发原因、主要风险、更新时间。

支持筛选、排序和导出 CSV/Excel。

### 个股详情

展示股票基础信息、行情指标、评分拆解、历史评分走势、近期触发信号和风险提示。

### 盯盘和提醒

用户可以为股票池或单只股票设置提醒：

- 价格达到阈值
- 涨跌幅达到阈值
- 综合评分变化达到阈值
- 技术指标突破

第一版使用准实时刷新，刷新频率由数据源和部署环境决定，不承诺毫秒级实时行情。

盯盘默认以任务轮询实现。刷新间隔先设为 1 到 5 分钟可配置，具体下限取决于公开数据源限制。提醒先在系统内展示，邮件、短信、微信等外部推送放到后续版本。

### 回测

用户选择股票池、策略版本、历史区间和回测参数后运行回测。

第一版回测参数包括：

- 调仓频率
- 最大持仓数量
- 单只股票权重方式
- 止损和止盈
- 手续费

输出：

- 收益曲线
- 最大回撤
- 年化收益
- 胜率
- 交易次数
- 交易明细
- 持仓快照

第一版回测采用日线级别，不做分钟级撮合。成交价先使用调仓日收盘价或下一交易日开盘价的可配置模式。停牌、涨跌停、滑点等市场细节以参数或限制项记录，第一版不追求专业量化平台级精度。

## 技术架构

推荐架构：

- 前端：Next.js 或 React
- 后端：FastAPI
- 数据库：PostgreSQL
- 缓存和任务：Redis + 后台任务队列
- 数据分析：Python pandas/numpy
- 图表：前端图表库
- 部署：保持平台无关，后续可部署到云服务器、Render、Railway、Vercel 加后端服务等环境

前端推荐 Next.js，后端推荐 FastAPI。第一版如果追求开发速度，可采用一个仓库管理前后端代码；后续部署时再拆分服务。

## 服务边界

### 前端

负责页面展示、交互、筛选、图表和导出入口。前端不直接访问第三方行情接口，避免跨域、密钥泄露和数据标准不一致问题。

### 后端 API

负责身份认证、团队数据、股票池、策略、评分、提醒、回测和导出任务。

### 数据源适配层

封装公开数据源，输出统一数据结构。后续更换或增加数据源时，不影响评分引擎和前端。

第一版优先接入 A 股公开数据源，字段按可用性分层：

- 必需字段：股票代码、名称、日期、开盘价、最高价、最低价、收盘价、成交量、成交额
- 推荐字段：市盈率、市净率、总市值、流通市值、换手率、行业
- 可选字段：资金流、财务摘要、指数成分、概念标签

如果公开数据源缺少基本面或资金面字段，评分引擎必须显示缺失原因，并按策略配置降低该分项权重或给出数据缺失风险提示。

Public Equity Investing 插件可以作为可选辅助数据源接入，用于补充基本面、估值、行业对比和风险解释。该插件不作为短线评分的唯一依赖；插件不可用时，系统仍应完成行情、量价、资金活跃度和风险项评分，只降低或跳过基本面排雷项。

数据源优先级：

- 主数据源：公开行情和量价数据，用于短线评分核心。
- 辅助数据源：Public Equity Investing 插件，用于基本面、估值、行业和风险解释。
- 备用数据源：同花顺导出、CSV/Excel、AkShare、Tushare 或其他可用公开接口。

所有数据源必须通过统一适配器输出标准字段，不允许评分引擎直接依赖某一个插件或第三方接口。

### 评分引擎

读取标准化行情、基础面、资金面和风险数据，按照策略版本计算分数并保存结果快照。

### 回测引擎

复用策略版本和历史数据，按用户配置生成调仓、交易、持仓和收益结果。

### 后台任务

处理定时行情刷新、评分任务、提醒检测、回测任务和导出任务。

后台任务需要支持任务状态查询，至少包含 queued、running、success、failed 四种状态。长任务不阻塞前端请求。

## 数据流

1. 用户创建或选择股票池。
2. 用户创建或选择策略版本。
3. 后端根据股票池拉取公开数据并缓存。
4. 数据源适配层标准化数据。
5. 评分引擎计算综合分和分项分。
6. 后端保存评分快照和触发原因。
7. 前端展示排行榜、详情和图表。
8. 提醒任务根据最新行情和评分变化触发提醒。
9. 回测引擎复用策略版本和历史数据生成回测报告。

## 评分模型

第一版使用短线优先的可解释综合评分，输出总分和分项分。评分目标不是判断长期投资价值，而是筛出短期活跃、趋势较强、流动性较好且风险可控的候选股票。

默认总分为 0 到 100 分。每个分项先标准化到 0 到 100，再按策略权重合成。风险项作为扣分项处理。评分结果必须保留主要贡献因子，避免只给黑盒分数。

技术面指标：

- 趋势强度
- 均线状态
- 成交量变化
- 突破信号
- 波动率
- 近 3 到 20 个交易日涨跌幅
- 关键价位突破或回踩

基本面指标：

- 估值水平
- 成长性
- 盈利能力
- 财务稳定性
- ST、退市风险、重大财务异常过滤

资金面指标：

- 成交额
- 换手率
- 资金活跃度
- 量比
- 近期成交额排名

风险扣分：

- 高波动
- 连续大涨
- 流动性不足
- 财务异常
- 数据缺失
- 短期涨幅过大
- 跌破关键均线
- 异常放量后回落

默认权重建议：

- 技术面：45%
- 资金面：35%
- 风险扣分：15%
- 基本面：5%

用户可以调整权重，但总权重需要归一化。策略保存后产生新版本，历史评分和回测不被新配置覆盖。

短线默认策略应提供至少三种模板：

- 强势突破：偏好放量突破、均线多头、成交额活跃的股票。
- 回踩反弹：偏好趋势仍在、短期回调到关键均线附近并出现企稳迹象的股票。
- 低位异动：偏好近期成交额和换手明显放大、但短期涨幅尚未过大的股票。

每只股票输出：

- 综合分
- 技术分
- 基本面分
- 资金分
- 风险扣分
- 主要入选原因
- 主要风险提示

## 核心数据模型

主要实体：

- User：用户
- Team：团队
- TeamMember：团队成员
- StockPool：股票池
- StockPoolItem：股票池成分
- Strategy：策略
- StrategyVersion：策略版本
- DataSourceSnapshot：数据源快照
- ScoreRun：评分运行记录
- StockScoreSnapshot：股票评分快照
- WatchlistAlert：提醒配置
- AlertEvent：提醒事件
- BacktestRun：回测运行记录
- BacktestTrade：回测交易明细
- BacktestPositionSnapshot：回测持仓快照
- BrokerageConnectionPlaceholder：实盘账户连接占位

关键关系：

- User 可以拥有多个 StockPool 和 Strategy。
- Team 可以拥有多个共享 StockPool 和 Strategy。
- Strategy 可以拥有多个 StrategyVersion。
- ScoreRun 绑定 StockPool 和 StrategyVersion。
- StockScoreSnapshot 绑定 ScoreRun 和股票代码。
- BacktestRun 绑定 StockPool、StrategyVersion 和回测参数。

## 错误处理

数据源失败时：

- 标记数据源状态
- 使用最近一次可用缓存
- 在结果中显示数据更新时间和缺失提示

评分失败时：

- 保存失败状态和错误原因
- 不覆盖上一次成功结果

回测失败时：

- 保存失败状态
- 展示失败阶段和可操作错误信息

提醒失败时：

- 记录失败事件
- 不重复发送同一触发条件的提醒

数据质量异常时：

- 对缺失字段做显式标记
- 对异常价格、停牌、零成交量做风险提示
- 对超过缓存有效期的数据禁止作为最新评分依据

权限错误时：

- 个人资产只允许所有者访问
- 团队资产只允许团队成员访问
- 未授权访问返回明确错误，不泄露资产名称或明细

## 安全和合规边界

第一版不处理真实交易凭证，不连接券商实盘账户，不执行下单。

用户密码必须加密存储。后续如果加入实盘账户连接，必须单独设计凭证加密、最小权限、审计日志、人工确认、风控限制和模拟盘验证。

第一版需要实现：

- 密码哈希存储
- 登录态过期
- 基础请求鉴权
- 用户和团队数据隔离
- 后端接口参数校验
- 任务和错误日志

页面上需要明确提示：评分和回测结果仅用于研究分析，不构成投资建议。

## 非功能要求

- 常规页面响应应保持在 2 秒内。
- 评分和回测等长任务可以异步完成，并在页面展示进度或状态。
- 系统应能处理至少 100 只股票的单次评分任务。
- 数据源失败时，系统应展示可理解的错误信息，而不是空白页面。
- 导出文件需包含数据更新时间、策略版本和评分运行编号。
- 主要业务对象需要创建时间、更新时间和所有者信息。

## 验收标准

第一版完成后应满足：

- 用户可以注册、登录并进入仪表盘。
- 用户可以创建个人股票池和团队股票池。
- 用户可以创建团队并通过邀请链接或邀请码添加成员。
- 用户可以创建策略并调整评分权重。
- 系统提供至少三种短线策略模板：强势突破、回踩反弹、低位异动。
- 系统可以从公开数据源拉取至少一组可用行情或基础数据。
- 用户可以运行评分并看到综合排行榜。
- 每只股票可以查看评分拆解和历史评分。
- 用户可以设置至少一种提醒并看到提醒事件。
- 用户可以运行一个日线级历史区间回测，并看到收益、回撤、胜率和交易明细。
- 用户可以导出选股结果。
- 导出结果包含数据更新时间、策略版本和评分运行编号。
- 系统不提供自动交易、券商下单或真实实盘账户连接功能。
